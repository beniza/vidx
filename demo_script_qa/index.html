    VIDX Demo Script — Ground Truth Q&A

    Prep notes for a 12-minute live demo aimed at Scripture translation/literacy teams who have already built a Scripture App for their language using Scripture App Builder (SAB). Answers are grounded in the actual code and docs (ffmpeg_builder.py, usfm_parser.py, youtube.py, getting_started.md, user_guide.md, session_summary.md, code_health_report.md, and real SAB-format timing files under src/) as of commit 6e8dd3f — not marketing copy.

    1. What you do

    VIDX takes the USFM scripture text, audio recording, and timing file your team already generated while building your Scripture App in Scripture App Builder, and turns them into a finished MP4 video with word/verse-synced on-screen subtitles, ready to upload to YouTube. The hard part — aligning scripture text to the audio — is work your team has already done for SAB. VIDX doesn't re-do that alignment; it reuses SAB's existing timing output and repurposes it for YouTube, with minimal extra effort.

    2. Inputs

    Three files — all artifacts your team already has (or already knows how to produce) from your SAB project, none generated by VIDX itself:

    • USFM scripture file (.SFM/.usfm, plain text with standard markers like \c, \v) — the same text used in your Scripture App. Common mistake: non-standard or missing chapter markers, so VIDX can't detect which chapter it's looking at.
    • Timing file (.txt, tab-separated start_time \t end_time \t segment_id, with header lines \id, \c, \level, \separators) — this is the exact audio-sync timing file Scripture App Builder produces, already sitting in your SAB project folder. VIDX does not create or re-align timing; it consumes SAB's output as-is. Common mistake: the file naming convention (SAB exports something like C01-41-MRK-01-timing.txt) not matching what VIDX expects for chapter-number detection, or a timing file from a different chapter than the USFM/audio it's paired with.
    • YAML config file (font, colors, background video/image, resolution, watermark). Most common mistake, per the project's own FAQ: specifying a font (e.g. Nirmala UI, Mangal, Bailey) that isn't actually installed on that machine — produces square "tofu" boxes instead of text, with no advance warning.

    3. Happy path (per chapter, once SAB files exist)

    1. Open a terminal in the project folder — manual, ~10 sec.
    2. Optionally edit the .yaml config in Notepad — manual, 0-5 min.
    3. Run vidx.exe -c yourbook.yaml [--gpu -w 4] — automatic, ~55-75 sec/chapter on GPU (real production numbers; no hard measured number for CPU-only).
    4. VIDX writes the .mp4, a title-card thumbnail, and a YouTube_Upload_Package/ folder — automatic.
    5. Publish:
    6. Manual path: open YouTube Studio, drag in the .mp4, paste the pre-written title/description from metadata.txt, upload the thumbnail, publish — manual, a few minutes per video, no setup required.
    7. Automated path: run vidx --manifest output/publish_manifest.json — automatic upload+thumbnail+playlist assignment, capped by Google at ~5-6 videos/day; re-run the same command the next day to resume.

    4. Prerequisites

    • Windows PC (the packaged vidx.exe is Windows-only right now — no Mac/Linux build exists yet, that's an open, undecided issue).
    • FFmpeg 4.3+ installed separately and on PATH, built with --enable-libass --enable-fontconfig --enable-libharfbuzz. Not bundled inside vidx.exe — a real, separate install step people often miss.
    • The fonts you intend to use, installed on that specific machine.
    • For automated YouTube upload only: a free Google Cloud project, YouTube Data API v3 enabled, an OAuth "Desktop app" client secret downloaded to ~/.vidx/client_secrets.json, and your email added to the app's OAuth Test users list (miss this and you get Error 403: access_denied). No cost at typical volumes — the real constraint is Google's daily quota, not billing.

    5. Output

    An MP4 (16:9 widescreen or 9:16 vertical) with the background video/image looping under the narration audio, scripture text burned in as styled subtitles (color/outline/position all configurable), an optional logo watermark, and optional background music auto-ducked under the voice. Complex scripts — Devanagari, Arabic-derived, Malayalam, Thai — are supported via HarfBuzz shaping and have been used in real production (Sindhi). Honest gap: true right-to-left layout (Arabic, Urdu, Hebrew specifically) has no dedicated test coverage and is an open, unresolved issue — nobody has yet confirmed what, if anything, is actually broken vs. just untested.

    6. Time & reliability

    A single chapter is roughly 1 minute of GPU render time; a full book (20-30 chapters) with 4 parallel workers took on the order of an hour in real production, though 4-way parallelism doesn't scale linearly (GPU encoder contention nearly doubles per-chapter time). The YouTube upload step is the flakiest/slowest part on weak internet — though it uses Google's resumable chunked upload, so a dropped connection resumes rather than restarts. The render path itself has zero automated end-to-end test coverage (CI runs unit tests only, no ffmpeg installed in CI) — a regression here could ship silently.

    7. Honest limitations

    • VIDX trusts the SAB timing file completely and has no automated check for a timing row silently mapped to the wrong verse/segment — a bad SAB export renders silently wrong, with no warning (an outlier-detection feature is an open, unimplemented issue).
    • No resume-without-rework: an interrupted batch currently re-renders every chapter from scratch, even ones that already succeeded (confirmed to waste ~50-60 min of GPU time in the last production run; fix not yet built).
    • YouTube's daily upload quota (~5-6 videos) is a hard external ceiling VIDX cannot lift, and VIDX's own quota counter forgets state between process restarts, so it's possible (if rare) to think you have quota left when you don't. This is the thing most likely to embarrass a live demo — pushing more than ~5 real uploads back-to-back will hit Google's wall on stage.

    8. Easiest entry (under 10 minutes, first try)

    Run one of the provided example configs (a single already-fully-set-up chapter with its USFM/timing/background bundled in examples/) with vidx.exe -c examples/<sample>.yaml — no new files to prepare, no YouTube account needed, and you have a finished, subtitle-synced MP4 in under 2 minutes to play back locally.

    9. "All you need is ___"

    "...if you've already built a Scripture App in Scripture App Builder, you've already done the hard work — VIDX just takes that same text, audio, and timing file and turns it into a YouTube video."

    10. What's in the pipeline

    Grounded in the project's open GitHub issue tracker (19 open issues as of this writing) — good for answering "what's next?" honestly, without over-promising dates.

    Near-term fixes (scoped, no open design questions): - Skip re-rendering chapters that already succeeded when a batch is restarted — directly addresses a real ~50-60 min GPU-time waste hit in production. - Persist YouTube's daily quota usage across restarts, so VIDX's own safety check can't be fooled by a process restart into overrunning Google's real limit. - YouTube chapters generated automatically from existing section headings, and verse-level timestamps added to the video description. - Small reliability/polish items: clean Ctrl+C cancellation, removing a cosmetic startup warning, hardening the Windows launcher script.

    Being explored (open questions, not yet scoped for build): - Song videos with karaoke-style word highlighting — extending VIDX beyond scripture chapters to worship/gospel songs, with a synced translated "meaning" subtitle track (e.g. English/Hindi) alongside the native-language lyrics. - A GUI config editor for field teams who'd rather not hand-edit YAML. - Right-to-left script rendering (Arabic, Urdu, Hebrew) — no confirmed bug yet, needs real test cases. - Vimeo as a second publishing target alongside YouTube. - A Linux build, if there's real demand for one. - AI-assisted config generation — pointing a chat-based AI assistant at an existing scripture folder and having it infer file-naming conventions to generate a first-draft project.yaml, instead of hand-writing it. Still produces a normal, inspectable YAML that runs through VIDX exactly as today — the AI only helps author it, it's not a new runtime dependency. Not ready to announce publicly yet — do not mention this one in the live demo.

    Honest framing for the demo: none of the "being explored" items have committed timelines — they're documented ideas, not promises. Only claim the near-term items as "coming soon," and even those as "in progress," not "next release."
